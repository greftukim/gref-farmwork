**B-2 핫픽스 경위 (2026-04-09 검증 중 발견):**
- 증상: 박민식 행에 대리 입력 시도 시 모달 "저장 중..." 상태에서 멈춤
- 콘솔: `Uncaught (in promise) RangeError: Invalid time value at Date.toISOString`
- Network: Supabase 요청 발생 안 함 (클라이언트 단계 실패)
- 원인: Supabase TIME 타입이 'HH:MM:SS' (예: '10:42:00') 반환 → prefill로 time
  input에 그대로 들어감 → handleProxyConfirm에서 ':00' 이중 부착 →
  'T10:42:00:00' → Invalid Date → RangeError
- 수정 (AttendancePage.jsx):
  - prefill 단계에서 `toHHMM(t) = String(t).slice(0, 5)` 헬퍼로 정규화
  - Date 생성 시 `.slice(0, 5)` 이중 방어
- 재현 조건: prefill 값 수정 없이 그대로 저장 시 100% 재현. time input을 한 번이라도
  수정하면 브라우저가 HH:MM으로 정규화해 우회 가능.
- **교훈:** Supabase PostgreSQL TIME 타입의 반환 형식을 지시문에서 사전에 명시하지
  못한 것. 타입 변환 이슈를 설계 단계에서 검토해야 함.

```js
   export function judgeAttendanceStatus(checkInTime, workStartTime) {
     if (!workStartTime) return 'normal';
     let hhmm = checkInTime;
     if (checkInTime.includes('T')) {

       hhmm = `${String(d.getHours()).padStart(2, '0')}:${String(d.getMinutes()).padStart(2, '0')}`;

     const standard = String(workStartTime).slice(0, 5);
```

```sql
CREATE TABLE public.safety_checks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  worker_id UUID NOT NULL REFERENCES public.employees(id) ON DELETE CASCADE,
  date DATE NOT NULL,
  check_type TEXT NOT NULL CHECK (check_type IN ('pre_work', 'post_work')),
```

## 이번 세션 교훈

### 1. 스키마 조사를 db-schema.md에 의존하지 말 것

**사건:** growth_surveys 확장 마이그레이션(20260409140000) 작성 → 적용 → 그 다음
코드 조사 중 `measurements` JSONB 컬럼이 이미 존재하고 활발히 사용 중임을 발견
→ 롤백 + git reset.

**교훈:** 마이그레이션 작성 전 해당 테이블의 현재 스키마를 **DB에서 직접 확인**.
`information_schema.columns` 조회. db-schema.md는 실제 DB보다 낡았을 수 있음.

### 2. 기능 추가 전 grep 필수

**사건:** 동일. 엑셀 양식에서 항목 뽑아 9개 컬럼을 새로 추가했으나 기존
`measurements` JSONB + `growth_survey_items` 마스터 테이블 기반의 동적 측정값
시스템이 이미 있었음.

**교훈:** 새 컬럼/테이블 추가 전 `grep -rn "<핵심 키워드>"` 실행. 특히 기존
stores/pages에서 비슷한 목적의 코드가 있는지 확인.

### 3. Supabase TIME 타입 주의

**사건:** B-2 핫픽스. Supabase의 `TIME` 컬럼은 'HH:MM:SS' 형식으로 반환되는데
HTML `<input type="time">`은 'HH:MM' 형식. prefill 시 형식 불일치 → Date 생성
실패 → RangeError.

**교훈:** Supabase 타입별 JavaScript 반환 형식을 설계 단계에서 확인. 지시문에
타입 변환 이슈를 명시. time input과 관련해서는 항상 `String(value).slice(0, 5)`
방어.

### 4. 스코프 통제의 어려움

**사건:** 이 세션에서 트랙 A/B 마무리 중 "tasks 시드 추가" → "growth_surveys 시드
추가" → "엑셀 양식 반영 스키마 확장" → "생육조사 그래프(트랙 C)" → "TBM 안전점검
(트랙 D)" → "관리자 조회 페이지까지" 요청이 연속으로 들어와 스코프가 비선형적으로
확장. 중간에 롤백 1회 + 옵션 c2 협의 + 다시 축소 결정.

**교훈:**
- 세션 초반에 스코프를 명시하고 새 요청은 인수인계 문서에 기록하는 습관
- "꼬리 물기"가 시작되면 한 번 멈추고 현재 범위 재확인
- 요청이 들어올 때 "이건 이번 세션인가, 다음 세션인가?" 먼저 질문
- 태우가 직접 2026-04-09 세션 후반에 스코프 통제를 명시적으로 요청함 — 이 결정이
  세션 종료를 정돈시킴

### 5. Claude Code 덤프가 잘리는 경우 대응

**사건:** 트랙 A 작업 중 Claude Code가 큰 파일 내용을 출력할 때 중간에 truncate
되는 경우 발생. 전체 내용이 필요했으나 일부만 받음.

**교훈:** 큰 파일 덤프 요청 시 좁은 범위로 나누거나 view의 view_range 활용.
필요 시 반복 요청.
